# 🌾 Subsidy Scoring — Decentrathon 5.0 / Кейс 2

Система скоринга заявок на субсидии АПК Казахстана на основе машинного обучения.  
Merit-based подход вместо принципа «первый подал — первый получил».

## 📋 Что делает система

- **Ранжирует** заявителей по вероятности одобрения (score 0–100)
- **Объясняет** каждое решение через SHAP waterfall (какие факторы повлияли)
- **Формирует shortlist** кандидатов с трёхзонной рекомендацией:
  - ✅ **Авто-одобрение** — высокий приоритет, p ≥ 0.807
  - ⚠️ **Ручная проверка** — передать комиссии, 0.55 < p < 0.807
  - ❌ **Авто-отклонение** — низкий приоритет, p ≤ 0.55

## 🧠 Логика ML модели

### Постановка задачи

**Цель:** предсказать вероятность одобрения заявки на субсидию АПК.

**Целевая переменная (v5):**
```python
# Позитивный класс (target = 1):
{"Исполнена", "Одобрена", "Сформировано поручение", "Получена"}  # 86.4%

# Негативный класс (target = 0):
{"Отклонена", "Отозвано"}  # 13.6%
```

Выбор такого target обоснован тем, что "Одобрена" — заявка прошла комиссию и ждёт выплаты. Предсказывать только "Исполнена" — значит штрафовать области где выплаты ещё не завершены (проблема Алматинской, Жамбылской областей в v4).

### Архитектура

```
Данные заявки
      ↓
Feature Engineering (33 признака)
      ↓
┌─────────────┐   ┌─────────────┐
│  LightGBM   │   │  CatBoost   │
│  5-fold OOF │   │  5-fold OOF │
└──────┬──────┘   └──────┬──────┘
       └────────┬─────────┘
           Meta LogReg (стекинг)
                ↓
        Isotonic Calibration
                ↓
         P(одобрения) ∈ [0, 1]
                ↓
         Трёхзонное решение
```

### Признаки (33 шт.)

**Временны́е (6):** месяц, день_года, день_недели, час, квартал, пиковый_сезон

**Финансовые (7):** log(сумма), log(норматив), log(поголовье), сумма/норматив,
кратность нормативу, остаток от деления, децильный ранг суммы

**Порядковый ранг (1):** log(ранк заявки в день по акимату) — показывает эффект
«первый подал — первый получил»

**Target encoding (6):** средняя успешность по области / акимату / району /
типу_субсидии / направлению / (область × направление) — считается внутри каждого
OOF фолда, нет утечки данных

**Frequency encoding (2):** частота встречаемости области и типа субсидии

**Относительные суммы (2):** сумма vs медиана по области, сумма vs медиана по типу

**Внешние агрегаты (9):** из данных акиматов 17 областей (data.egov.kz) —
историческая успешность, средняя сумма выплат, активность области

### Оптимизация

Optuna 50 trials оптимизирует `Precision @ Recall ≥ 85%` — максимизируем точность
при условии что охватываем минимум 85% достойных заявок.

```python
# Бизнес-логика метрики:
# FP = одобрили недостойного  → дорогая ошибка (бюджет)
# FN = отклонили достойного   → комиссия пересмотрит
```

### Результаты (v5)

| Метрика | Значение |
|---|---|
| P@Recall≥85% | **0.9141** |
| ROC-AUC | 0.7746 |
| PR-AUC | 0.9526 |
| Precision авто-одобрения | **91.4%** |
| Авто-одобрение | 80.4% |
| Ручная проверка | 15.4% |
| Авто-отклонение | 4.3% |

### Объяснимость (SHAP)

Топ признаки по важности:
1. `te_district` — историческая успешность района
2. `te_subsidy_type` — историческая успешность типа субсидии
3. `te_oblast_dir` — успешность (область × направление)
4. `day_of_year` — сезонность одобрений
5. `log_daily_rank` — порядок подачи в день
